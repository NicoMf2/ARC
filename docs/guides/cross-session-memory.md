# Cross-Session Memory Guide

ArcKit includes a cross-session memory system that captures key decisions, vendor insights, research findings, and session summaries across Claude Code sessions. This allows Claude to recall what happened in previous sessions and build on prior work rather than starting from scratch.

> **Bundled Integration**: The memory MCP server is included in the ArcKit plugin and requires no additional configuration beyond having Node.js installed.

---

## How It Works

The memory system has two parts:

1. **Session Learner** (`hooks/session-learner.sh`) — fires when a session ends, analyses recent git commits, and writes a JSON manifest summarising what happened
2. **Session Start** (`hooks/arckit-session.sh`) — fires when a new session begins, reads pending manifests, and surfaces them as context instructing Claude to store them via the MCP memory server

This two-stage approach is necessary because **MCP tools are not available inside hooks** (hooks are Bash scripts, not Claude tool calls). The session learner writes manifests to disk; the session start hook reads them and presents them to Claude for processing.

### Data Flow

```
Session N ends
  └── session-learner.sh analyses git commits
       └── writes .arckit/memory/pending-{session_id}.json

Session N+1 starts
  └── arckit-session.sh reads pending manifests
       └── surfaces context: "Previous session created REQ v1.0..."
            └── Claude calls mcp__memory__create_entities
                 └── entities stored in .arckit/memory.jsonl
                      └── Claude calls mcp__memory__search_nodes
                           └── retrieves relevant prior context
```

## Entity Types

The memory system uses seven entity types designed for architecture governance workflows:

| Entity Type | What It Captures | Example Observations |
|---|---|---|
| `ProjectDecision` | Key ADR outcomes that should persist | `["chose-kafka", "rejected-rabbitmq", "3yr-tco-£450k"]` |
| `VendorInsight` | Vendor evaluation findings | `["hidden-egress-costs", "no-uk-dc", "poor-gov-cloud-support"]` |
| `ReviewOutcome` | HLD/DLD review results | `["approved-with-conditions", "missing-dr-strategy"]` |
| `ResearchFinding` | Market observations | `["g-cloud-no-federated-identity", "all-vendors-use-oauth2"]` |
| `RecurringRequirement` | Patterns across projects | `["data-sovereignty-nfr-in-4-of-5-projects"]` |
| `LessonLearned` | Operational discoveries | `["sobc-exceeds-32k-tokens", "research-needs-req-doc-first"]` |
| `SessionSummary` | What happened in a session | `["created-req-v1", "3-adrs", "hld-review-pending"]` |

### Entity Naming Conventions

- **SessionSummary**: `session-YYYYMMDD-HHMM` (auto-generated by session learner)
- **ProjectDecision**: `decision-{project-id}-{short-description}` (e.g., `decision-001-kafka-over-rabbitmq`)
- **VendorInsight**: `vendor-{vendor-name}` (e.g., `vendor-aws`, `vendor-azure`)
- **ReviewOutcome**: `review-{artifact-type}-{date}` (e.g., `review-hld-20260220`)
- **ResearchFinding**: `research-{topic}` (e.g., `research-g-cloud-identity`)
- **RecurringRequirement**: `recurring-{pattern}` (e.g., `recurring-data-sovereignty`)
- **LessonLearned**: `lesson-{short-description}` (e.g., `lesson-sobc-token-limit`)

## Design Principles

### Memory Owns Learnings, Not Artifact Copies

The memory system stores **observations about decisions and outcomes**, not copies of artifacts. If you need the full content of an ADR, read the file. Memory captures what matters for cross-session continuity:

- "We chose Kafka because of the throughput requirement" (memory)
- Not: the entire ADR document text (that lives in the file system)

### Keep Entity Count Low

The `read_graph` MCP tool returns **all entities with no filtering**. The practical cap is approximately 50 entities before context window impact becomes noticeable. To stay within this limit:

- Use `SessionSummary` entities for automatic session tracking (prune after 30 days)
- Use specific entity types (`ProjectDecision`, `VendorInsight`, etc.) for persistent learnings
- Periodically consolidate related observations into fewer entities

### Project-Local Storage

The memory file lives at `.arckit/memory.jsonl` within the project directory, not globally. This means:

- Each project has its own memory — no cross-contamination
- Memory travels with the project (committed or gitignored, your choice)
- Different team members can share project memory via version control

## Session Classification

The session learner automatically classifies sessions based on which artifact types were created or modified:

| Session Type | Triggered By | Purpose |
|---|---|---|
| `governance` | ADR, PRIN, STRAT, DPIA, JSP936 | Architecture decision-making |
| `research` | RSCH, AWRS, AZRS, GCRS, DSCT | Technology or market research |
| `procurement` | SOBC, SOW, EVAL, DOS, GCLD, GCLC | Procurement and commercial work |
| `review` | HLDR, DLDR, TCOP, SECD, SECD-MOD | Design reviews and compliance |
| `general` | Any other artifacts | General architecture work |

## MCP Memory Tools

The memory MCP server provides these tools (auto-allowed by the `allow-mcp-tools.sh` hook):

| Tool | Purpose | When to Use |
|---|---|---|
| `mcp__memory__create_entities` | Store new entities with observations | After processing a pending manifest |
| `mcp__memory__add_observations` | Add observations to existing entities | When new findings relate to existing entities |
| `mcp__memory__search_nodes` | Search entities by query | Before starting work — check for prior context |
| `mcp__memory__read_graph` | Read all entities | Periodic review of stored knowledge |
| `mcp__memory__create_relations` | Link entities together | When decisions depend on vendor insights, etc. |
| `mcp__memory__delete_entities` | Remove stale entities | Pruning old session summaries |
| `mcp__memory__delete_observations` | Remove specific observations | Correcting outdated information |

## Configuration

The memory MCP server is configured in the plugin's `.mcp.json`:

```json
{
  "memory": {
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-memory"],
    "env": {
      "MEMORY_FILE_PATH": ".arckit/memory.jsonl"
    }
  }
}
```

### Prerequisites

- **Node.js 18+** — required to run the MCP server via `npx`
- **npm** — `npx` downloads `@modelcontextprotocol/server-memory` on first use

### .gitignore Considerations

You may want to add the memory files to `.gitignore` if they contain sensitive project information:

```gitignore
# Cross-session memory (optional — exclude if sensitive)
.arckit/memory.jsonl
.arckit/memory/
```

Alternatively, commit them to share project memory across team members.

## Manual Memory Operations

You can ask Claude to create memory entities directly during a session:

```
Store a project decision: we chose AWS over Azure for the data platform
because of the existing S3 data lake and £120k/year savings on egress.
```

Claude will call `mcp__memory__create_entities` with:

```json
{
  "entities": [{
    "name": "decision-001-aws-data-platform",
    "entityType": "ProjectDecision",
    "observations": [
      "chose-aws-over-azure",
      "existing-s3-data-lake",
      "120k-annual-egress-savings"
    ]
  }]
}
```

You can also query memory at any time:

```
What vendor insights do we have from previous sessions?
```

Claude will call `mcp__memory__search_nodes` with query `"vendor"` and summarise the results.

## Pending Manifest Format

The session learner writes manifests to `.arckit/memory/pending-{session_id}.json`:

```json
{
  "timestamp": "2026-02-20T14:30:00Z",
  "session_id": "abc123",
  "entity": {
    "name": "session-20260220-1430",
    "entityType": "SessionSummary",
    "observations": [
      "session-type-governance",
      "3-commits",
      "created-adr",
      "created-req",
      "worked-on-Architecture-Decision-Record",
      "worked-on-Requirements",
      "feat: add ADR for Kafka selection",
      "feat: update requirements with throughput NFR"
    ]
  }
}
```

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| No memory context on session start | No pending manifests | Session learner only fires on stop; check `.arckit/memory/` |
| `npx` fails to start memory server | Node.js not installed | Install Node.js 18+ |
| Memory file grows too large | Too many session summaries | Delete old `SessionSummary` entities via `mcp__memory__delete_entities` |
| Memory entities not persisting | Wrong `MEMORY_FILE_PATH` | Check `.mcp.json` — path must be relative to project root |
| Session learner not firing | Hook not registered | Verify `Notification` event in `hooks.json` |
| `read_graph` returns too much | Exceeded ~50 entity cap | Consolidate or prune entities |
